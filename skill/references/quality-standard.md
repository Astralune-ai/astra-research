# research 质量标准（强制 gate）

> 质量 = **信息获取能力 × 采集工作流 × 信息审核判断**，三者缺一不可。
> 获取能力见 [`source-registry.md`](source-registry.md)；本文件管工作流（A、B、D）与判断（C）。

## A. 研究计划 gate（fan-out 前必经）

**这是"统一标准"所在——别从 Scope 直接 fan-out 子 agent。**

**强制度（Asher 定，2026-06-10）：**
- **大项目 / 高风险 / 对外交付** → **停下来**，写研究计划给 Asher 过，批准后才 fan-out。
- **小调研 / 内部快查** → 自动跑，但研究计划**仍落盘**（并入 `00_scope.md`），事后可查。
- 判别「大/小」：涉及投资·合规·对外交付·标的金额大·结论驱动重大决策 = 大；单一问题快查 / 内部 primer = 小。拿不准按大处理。

**研究计划内容**（写进 `00_scope.md` 的「研究计划」节）：

| 研究问题 | 拆成的研究线 | 必打源层级（Tier-1 优先名单） | 用哪个源 | 验收标准 |
|---|---|---|---|---|

- **验收标准默认**：关键数字 ≥2 个独立 Tier-1/2 源互证才算「查到」；查不到 → 标缺口，**不放 Tier-3 凑数**。
- 附 **已知头部源/玩家名单**（给子 agent 去验证存在性，比从 0 找准）。

## B. 子 agent 源纪律 prompt 模板（每个采集 agent 必带，不靠临场自觉）

```
你是「<研究线>」数据猎手，为 <视角：内部DD怀疑者 / 说服者 / 中立> 做带引用研究。不许编造。

## 背景
<标的速写 + 本线要压测的具体 claim/数据>

## 研究问题
<本线 2-4 个问题>

## 源纪律（硬）
- 优先 Tier-1（政府/监管/官方规格/公司 IR/学术/官方名单）；关键数字标 URL + 标来源 Tier。
- Tier-3（SEO 报告作坊 / 聚合站 / 无署名博客）只当「线索」去验证，**不当事实**。
- 三角验证：一个数字要 ≥2 个独立 Tier-1/2 源互证；分歧大 → 报区间 + 标红旗（分歧本身就是发现）。
- 自报/vendor 数据一律对抗式拿独立规格压测，判定 可信/夸大/无法验证。

## 输出
结构化 Markdown（你的最终输出即文件正文）；关键数字标 URL+Tier；区分有来源 vs 估算；
结尾「置信度与缺口」一节。约 6-8 分钟，到点即返回，缺口标清。
```

## C. 信息审核判断 rubric（每条结论进报告前过）

1. **源分级** — Tier-1 一手/权威 ＞ Tier-2 可信二手（成熟行业媒体/具名分析师）＞ Tier-3 低信。Tier-3 不当事实。
2. **三角验证** — ≥2 独立 Tier-1/2 互证；分歧 → 区间 + 红旗。
3. **自报数据压测** — 标的/vendor 自己给的，独立核 → 可信/夸大/无法验证。
4. **相关性** — 这数字会不会改变结论？不改变的不进正文。
5. **置信标注** — 每个关键结论标 高/中/低 + 缺口；估算标「估算」；查不到标「缺口」；**绝不编可被 fact-check 的业务事实**。
6. **对抗式 Verify** — 派独立怀疑者专门去「推翻」关键论断，推不翻才留（质量最后一道闸；委托方明确说"验证先放一边"时可暂缓，缺口集中进 REPORT 待核实节）。

## D. 覆盖度检查（Collect 后、Assemble 前）

- scope 每个研究问题是否都有答案？
- 有没有该跑没跑的源/模态（漏了 astra-probe？漏了制裁名单？）？
- 哪些落「缺口」→ 集中进 REPORT 待核实节，别藏。

## E. 报告站位与结构（Assemble 用）
结构、站位、配图规则**全部以 [`report-template.md`](report-template.md) 为唯一真源**（顾问综述前置 + 0–8 节；中立站位不替委托方拍板；主章一–六配图、收尾章豁免）。本文件不复述，以免两处过期。

---
**一句话**：标准强制质量，不靠子 agent 临场自觉。每跑一次 research，A→B→（采集）→D→C 走全 + 按 E 站位成稿，大项目 A 停下来给 Asher 过。
